# 서비스 고도화 기획자료 — 전체 계획 및 진행 가이드

> **목적**: Part A → B → C → D → 부록을 **컨텍스트 길이를 고려해 파트별로** 진행할 수 있도록, 순서·의존관계·맥락 이어짐·입출력을 상세히 정리한다.  
> **사용법**: 이 문서로 전체 흐름을 파악한 뒤, [01_파트별_프롬프트_문서.md](01_파트별_프롬프트_문서.md)의 프롬프트를 복사해 각 파트를 순서대로 실행한다.

| **01_파트별_프롬프트_문서.md** | 파트별 복사용 프롬프트 | 프롬프트 복사 시 참조 |

### 1.2 진행 순서와 맥락 이어짐

```
[기준 문서들]                    [파트 문서]
  PRD, development_plan,    →   Part A (기능·개념)
  development_progress,              ↓
  competitor_analysis               Part A 산출 요약
                                        ↓
                                    Part B (고도화·로드맵)
                                        ↓
                                    Part B 산출 요약
                                        ↓
                                    Part C (기획 대표·CTO 준비)
                                        ↓
                                    Part C 산출 요약
                                        ↓
                                    Part D (확장 실행)
                                        ↓
                                    부록 (용어·링크·시장 요약)
```

- **맥락 이어짐 원칙**: 각 파트 문서 끝에 **"핵심 산출 요약"** 섹션을 둔다. 다음 파트 실행 시 "이전 파트 문서 전체" 대신 **"이전 파트의 핵심 산출 요약"**만 읽어도 맥락이 유지되도록 한다.
- **컨텍스트 절약**: Part B만 할 때는 `part_b_*.md` + `part_a_*.md`의 "Part A 핵심 산출 요약"만 열면 되고, Part A 전체 본문을 반복해서 넣지 않아도 된다.

## 2. 파트별 상세 계획

### 2.1 Part A — 서비스 개요 및 현재 기능

| 항목 | 내용 |
|------|------|
| **목표** | 현재 구현 기능(As-Is)과 미구현 항목을 정리하고, 고도화 시 일관 쓰일 용어·개념을 정의한다. |
| **입력** | `docs/development_progress.md`, `docs/zodiac_service_prd.md`, 실제 `src/app` 디렉터리 구조. |
| **산출** | Part A 문서 내: (1) 기능 목록 표, (2) 미구현 항목 표, (3) 용어 정의(지표·서비스·점성술), (4) 서비스 핵심 개념(현재→고도화), (5) **Part A 핵심 산출 요약** (다음 파트용). |
| **다음 파트에 넘길 맥락** | 구현 완료 목록, 미구현 목록, 확정 용어, "현재=템플릿/규칙 기반, 고도화=트랜짓·AI" 개념. |
| **참조 문서** | development_progress.md, zodiac_service_prd.md. (컨텍스트 부족 시 해당 섹션만 인용) |

---

### 2.2 Part B — 고도화 방향 및 로드맵

| 항목 | 내용 |
|------|------|
| **목표** | Part A의 미구현 항목을 우선순위·선행 조건·담당·시기와 연결하고, 비기능 고도화·로드맵 다이어그램·단계별 배달물을 정리한다. |
| **입력** | Part A 문서의 **"Part A 핵심 산출 요약"** (필요 시 Part A 전체). PRD Phase 2/3, development_progress "다음 단계". |
| **산출** | Part B 문서 내: (1) 기능 고도화 우선순위 표, (2) 고도화 개념 요약(3축), (3) 비기능 고도화 표, (4) 로드맵 다이어그램(mermaid), (5) 단계별 핵심 배달물 표, (6) **Part B 핵심 산출 요약**. |
| **다음 파트에 넘길 맥락** | 우선순위·선행 조건·Phase별 배달물·고도화 3축(콘텐츠/참여/수익). |
| **참조 문서** | Part A, development_progress.md, zodiac_service_prd.md (로드맵·Phase). |

### 2.3 Part C — IPO/투자자 관점 준비사항

| 항목 | 내용 |
|------|------|
| **목표** | 기획 대표(서비스 기획)와 CTO가 투자자·실사 대응 시 준비할 항목을 역할별로 정리하고, 공통 "현재 상태 한 장 요약"과 "12개월 마일스톤"을 고정한다. |
| **입력** | Part A **핵심 산출 요약** (기능·개념), Part B **핵심 산출 요약** (로드맵·배달물). PRD(시장·수익·KPI·리스크), competitor_analysis(차별화). |
| **산출** | Part C 문서 내: (1) 기획 대표 준비사항(시장·지표·로드맵·리스크·규제), (2) CTO 준비사항(기술·확장성·보안·팀), (3) 공통(현재 상태 표·마일스톤), (4) **Part C 핵심 산출 요약**. |
| **다음 파트에 넘길 맥락** | 기획/CTO 각 산출물 예시(요약서 페이지 수), 공통 현재 상태·마일스톤. |
| **참조 문서** | Part A, Part B, zodiac_service_prd.md, competitor_analysis.md. |

### 2.4 Part D — 서비스 확장 준비 (기술·운영)

| 항목 | 내용 |
|------|------|

| **입력** | Part B **핵심 산출 요약** (비기능 고도화·DB 도입 시점), Part C **핵심 산출 요약** (CTO 기술·확장성). |

### 2.5 부록 — 용어·문서 링크·시장 요약

| 항목 | 내용 |

## 3. 실행 시 유의사항

### 3.1 컨텍스트 길이 대응

- **한 번에 한 파트만** 열어서 작업한다. 해당 파트 문서 + "이전 파트의 핵심 산출 요약"만 포함하면 대부분 맥락이 유지된다.
- 이전 파트 **전체 본문**이 꼭 필요할 때만 해당 파트 파일을 추가로 연다.
- 참조 문서(PRD, development_progress 등)는 **필요한 섹션만** 인용하거나 요약해 프롬프트에 넣을 수 있다.

### 3.2 수정 시 연쇄 반영

- **Part A** 수정(기능 추가/삭제, 용어 변경) → Part B 우선순위·고도화 개념, Part C 현재 상태·용어, 부록 용어에 반영 필요.
- **Part B** 수정(로드맵·배달물) → Part C 로드맵·마일스톤, Part D 확장 시점에 반영 필요.
- **Part C** 수정(마일스톤·준비 항목) → Part D 운영·확장 우선순위에 반영 필요.
- **통합본** `docs/service_advancement_plan.md`는 파트 문서 수정이 안정된 뒤, Part A–D·부록 순으로 병합해 갱신한다.

### 3.3 상태 표시

각 파트 문서 상단에 상태·최종 갱신일을 두고, 단계별 진행 후 갱신한다.
